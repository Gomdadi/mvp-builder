## 1. In-scope (MVP에 포함)

| ID | 기능명 | 포함 이유 | 설명 |
|----|--------|----------|------|
| F-01 | 자연어 요구사항 입력 | 서비스의 존재 근거. 없으면 서비스 자체가 성립 안 됨 | 단일 텍스트 영역, 최대 10,000자. 필수 입력값. |
| F-02 | MVP 프로젝트 자동 생성 | 핵심 가치 명제 그 자체 | Claude Agent SDK 파이프라인. 분석 → 피드백 대기 → 개발 → 테스트 → 업로드 순차 실행. |
| F-02a | 분석 문서 생성 및 반환 | 투명성 원칙 (C-VAL-02) + 개발자가 방향 확인 후 진행하는 신뢰 확보 | analyzing 단계 완료 후 ERD, API 설계, 아키텍처 요약을 UI에 표시. |
| F-02b | 사용자 피드백 제출 | 잘못된 방향으로 코드 생성되는 것을 사전 방지 | 분석 문서 검토 후 승인 또는 수정 요청. 피드백 제출 시 개발 단계 시작. |
| F-02c | 테스트 리포트 반환 | C-VAL-03 Reliability — 코드 품질 증명 | 테스트 실행 결과(통과/실패/커버리지 요약)를 clone URL과 함께 제공. |
| F-03 | 실시간 생성 진행률 표시 | C-VAL-02 Transparency 원칙. 블랙박스 처리 금지 | SSE 기반. 단계명 텍스트 + 전체 퍼센테이지. |
| F-04 | GitHub repo 자동 생성 및 업로드 | 결과물 전달의 유일한 수단 | 사용자 GitHub OAuth token으로 사용자 본인 계정에 공개 repo 생성, 전체 파일 커밋. repo명: `mvp-{keyword}-{username}` |
| F-05 | clone URL 제공 | C-VAL-03 Reliability — clone → install → run 보장 | 완료 직후 화면에 표시. 복사 버튼 제공. |
| F-06 | GitHub OAuth 로그인 | 생성 이력 저장 및 사용자 식별 + GitHub repo 생성 권한 동시 획득 | GitHub 계정으로 원클릭 로그인. 별도 이메일/비밀번호 불필요. JWT Access Token 발급. |
| F-07 | 생성 이력 조회 | clone URL은 재접근 가능해야 함 (C-AGENT-05) | 생성 목록 + 상태(분석중/피드백대기/개발중/완료/실패) + clone URL 재확인. |
| F-08 | 개발자 옵션 (선택 입력) | C-VAL-05 Extensibility + C-UX-01 Progressive Disclosure | 기본 접힘 상태. 기술 스택, 아키텍처, 배포 방식 입력 가능. UI: combobox 패턴 (선택지 + 자유 입력 혼합). |

## 3. MVP 핵심 가치 명제

### 가설 1 — 분석 문서 검토 후 생성된 코드베이스는 즉시 실행 가능하다

> "AI가 요구사항을 어떻게 해석했는지 개발자가 확인하고 피드백을 반영한 후 생성된 코드가, clone → install → run 이 오류 없이 동작하는 GitHub 저장소를 자동으로 생성할 수 있다."

- [결정] 생성 성공 기준: `npm run dev` 실행 시 오류 없이 정상 구동되어야 성공으로 판정
- 검증 기준: 생성된 저장소에서 `git clone → npm install → npm run dev` 오류 없는 실행 성공률 90% 이상 (내부 테스트 10회 이상 기준)

### 가설 2 — 피드백 루프가 단방향 생성 대비 결과물 만족도를 높인다

> "분석 문서를 먼저 보고 방향을 조정할 수 있는 개발자는, 바로 코드를 생성하는 경쟁 서비스 대비 결과물에 더 만족한다."

- 검증 기준: 생성 완료 후 테스트 사용자 만족도 70% 이상 (결과물을 실제 사용하거나 수정 의향 있음)

### 가설 3 — 테스트 리포트가 결과물 신뢰를 형성한다

> "clone 전에 테스트 통과 여부를 확인할 수 있으면, 개발자는 결과물을 더 신뢰하고 실제로 사용하는 비율이 높아진다."

- 검증 기준: clone URL을 실제로 사용(clone 또는 GitHub에서 열기)한 사용자 비율 60% 이상

## 4. 출시 전제 조건

MVP 출시를 위해 아래 조건이 모두 충족되어야 한다.

### 기능 완성도

- [ ] F-01 ~ F-08 (F-02a, F-02b, F-02c 포함) 모든 In-scope 기능 구현 완료
- [ ] 생성 파이프라인 (분석 → 피드백 대기 → 개발 → 테스트 → 업로드) 정상 동작
- [ ] 분석 문서가 awaiting_feedback 상태에서 UI에 정상 표시됨
- [ ] 피드백 제출 후 개발 단계가 올바르게 재개됨
- [ ] 테스트 리포트가 completed 이벤트와 함께 정상 반환됨
- [ ] SSE 이벤트가 각 단계 전환 시 정확히 클라이언트에 전달됨
- [ ] 사용자 GitHub 계정에 clone URL로 즉시 접근 가능한 repo가 생성됨

### 품질 기준

- [ ] Unit 테스트 핵심 비즈니스 로직 커버리지 80% 이상 (C-TEST-01)
- [ ] 전체 API 엔드포인트 Integration 테스트 100% 작성 (C-TEST-01)
- [ ] E2E 핵심 시나리오 3개 통과 (C-TEST-08, C-TEST-09, C-TEST-10)
- [ ] 생성 성공률 90% 이상 (내부 테스트 10회 이상 기준)

### 보안

- [ ] GitHub OAuth 로그인 흐름 완전 동작
- [ ] GitHub OAuth token AES-256 암호화 저장 확인
- [ ] JWT Access Token 15분 만료 동작 확인
- [ ] `.env` 파일 미포함, `.env.example` 포함 확인
- [ ] Claude API key 및 GitHub OAuth token 로그 미출력 검증

### 인프라

- [ ] Docker 컨테이너로 Backend/Frontend 패키징 완료
- [ ] `docker-compose.yml`로 로컬 전체 스택 실행 가능
- [ ] GitHub Actions CI 파이프라인 (lint + test) 동작 확인
- [ ] 프로덕션 AWS 배포 완료 및 HTTPS 적용
- [ ] CloudWatch Logs 연결 및 ERROR 레벨 로그 수집 확인

### 운영

- [ ] 운영자 Claude API key 환경 변수 등록 완료
- [ ] GitHub OAuth App 설정 완료 (Client ID, Client Secret, Callback URL)
- [ ] BullMQ + Redis 큐 프로덕션 환경 동작 확인
