# 성공 지표 및 KPI — AI 기반 자동화 MVP 빌더

---

## MVP 성공 기준

### North Star Metric

> **"파이프라인 완주율"**: 프로젝트를 생성한 사용자 중 GitHub 저장소 전달까지 완료한 비율
>
> 목표: 출시 후 1개월 이내 **70% 이상**

파이프라인을 완주했다는 것은 제품의 핵심 가치(실행 가능한 코드베이스 자동 전달)를 실제로 경험했음을 의미한다.

---

### Primary KPI

| # | 지표명 | 목표 수치 | 측정 기간 | Baseline |
|---|--------|-----------|-----------|----------|
| K-01 | 파이프라인 완주율 | 70% 이상 | 월간 | > 가정: 초기 0%에서 시작 |
| K-02 | 생성된 코드 `docker compose up` 성공률 | 90% 이상 | 파이프라인 완주 기준 | > 가정: 초기 측정값 없음 |
| K-03 | 월간 활성 사용자(MAU) | 출시 1개월 후 50명 이상 | 월간 | > 가정: 0명에서 시작 |

### Secondary KPI

## 지표별 측정 계획

| KPI | 측정 방법 | 측정 주기 | 담당 도구 |
|-----|-----------|-----------|-----------|
| K-01 파이프라인 완주율 | `projects` 테이블: `status = COMPLETED` / `status != CREATED` 비율 | 주간 | 자체 DB 쿼리 |
| K-02 Docker 성공률 | 별도 검증 Job: 생성 코드 빌드 테스트 결과 저장 | 파이프라인 완주 시마다 | 자체 로깅 |
| K-03 MAU | `users` 테이블: 월간 `pipeline_runs` 실행 있는 user 수 | 월간 | 자체 DB 쿼리 |
| K-04 확정률 | `analysis_documents`: `version = 1 AND is_confirmed = true` 비율 | 주간 | 자체 DB 쿼리 |
| K-05 완주 시간 | `pipeline_runs.started_at` ~ `projects.updated_at (COMPLETED)` 차이 | 주간 | 자체 DB 쿼리 |
| K-06 프로젝트/사용자 | `projects` COUNT GROUP BY `user_id` 월간 평균 | 월간 | 자체 DB 쿼리 |
| K-07 API Key 등록률 | `users`: `encrypted_api_key IS NOT NULL` / 가입 후 7일 경과 사용자 | 주간 | 자체 DB 쿼리 |

## 실패 기준 및 피벗 조건

### 실패 신호 (Month 1 기준)

| 신호 | 기준값 | 의미 |
|------|--------|------|
| 파이프라인 완주율 50% 미만 | K-01 < 50% | 코드 생성 품질 또는 UX 문제 |
| Docker 성공률 70% 미만 | K-02 < 70% | 생성 코드 실행 불가 → 핵심 가치 실패 |
| API Key 등록률 50% 미만 | K-07 < 50% | 온보딩 UX 문제 또는 비용 장벽 인식 |

### 피벗 조건

1. **K-02 Docker 성공률 저조**: 특정 기술 스택 조합으로 범위를 좁혀 품질 보장 (예: Next.js + NestJS 조합만 지원)
2. **K-01 완주율 저조**: Phase 별 이탈 지점 분석 후 해당 단계 UX/품질 개선 집중
3. **MAU 성장 정체**: 개발자 커뮤니티(Reddit, HackerNews) 타겟 마케팅 또는 무료 크레딧 프로모션

### Kill 기준

> 가정: Month 3 시점에 MAU 50명 미만이고 파이프라인 완주율 40% 미만인 경우 제품 방향 전면 재검토.
