## 1. 현재 상황 (As-Is)

### 비개발자의 현실

아이디어를 가진 비개발자(창업 준비생, 기획자, 도메인 전문가)는 자신의 서비스 아이디어를 코드로 구현하기 위해 다음 중 하나를 선택한다.

1. **외주/프리랜서 의뢰**: 최소 수백만 원의 비용, 수 주의 기간 소요. PoC 수준의 간단한 기능도 협의-계약-개발-납품 과정이 필요하다.
2. **노코드 도구(Bubble, Webflow 등)**: 시각적으로 앱을 구성할 수 있지만, 생성된 결과물이 해당 플랫폼에 종속되어 개발자에게 코드를 넘길 수 없다. 확장 시 플랫폼 한계에 부딪힌다.
3. **AI App Builder(Lovable, Bolt.new)**: 자연어 입력은 가능하지만, 크레딧 소진 시 작업이 단절되고, 생성 중 진행 상황을 알 수 없으며, 결과물을 어떻게 사용하는지(clone, 실행 방법) 안내가 부족하다.

> 가정: 비개발자의 AI App Builder 이탈 원인 1위는 "결과물을 어떻게 실행하는지 모른다"는 사용성 문제로 추정된다. 공식 데이터 미확인.

### 개발자의 현실

빠른 프로토타이핑이 필요한 개발자(주니어~시니어)는 새 프로젝트를 시작할 때마다 다음 과정을 반복한다.

1. CLI 도구(`create-next-app`, NestJS CLI 등)로 프로젝트 초기화 (10~30분)
2. ESLint, Prettier, TypeScript 설정 (30분~1시간)
3. 테스트 환경(Jest, Vitest) 구성 (30분)
4. Docker, GitHub Actions CI/CD 설정 (1~2시간)
5. 폴더 구조, 공통 모듈, 에러 핸들러 구성 (1~2시간)

보수적으로 잡아도 **반나절 이상의 반복 작업**이 아이디어 검증 전에 선행된다.

기존 AI 생성 서비스를 사용해도 기술 스택 선택지가 없거나(Lovable), 토큰 비용이 예측 불가하거나(Bolt.new), 생성된 코드가 실제 실행되지 않는 경우가 발생해 신뢰하기 어렵다.

### 기존 대안의 공통 한계 (경쟁사 분석 근거)

| 한계 | 해당 서비스 |
|------|------------|
| 생성 중 진행 상황 불투명 (스피너만 표시) | Lovable, Bolt.new, Replit Agent |
| 기술 스택 선택 불가, 특정 생태계 종속 | Lovable (Supabase/Vercel), v0 (React/Vercel) |
| GitHub 저장소 자동 생성 미지원 | Bolt.new, Replit Agent, v0 |
| 크레딧/토큰 소진 시 워크플로 단절 | Lovable, Bolt.new, v0 |
| 생성 결과물의 즉시 실행 불보장 | Bolt.new (복잡한 앱에서 오류 빈발), Lovable |
| 비개발자 접근 어려움 | GitHub Copilot Workspace |

## 2. 핵심 문제

### 문제 1 — 비개발자

> **아이디어를 가진 비개발자가**, 서비스를 데모 수준으로 구현하고 싶은 상황에서, **"즉시 실행 가능한 코드베이스를 자신이 이해할 수 있는 방식으로 받는 경로"가 없다.**

- 외주는 비싸고 느리다.
- 노코드는 개발자와 협업할 수 없는 종속된 결과물을 만든다.
- 기존 AI 생성 도구는 진행 상황이 불투명하고, 결과물 사용법이 낯설며, 크레딧 구조가 복잡하다.

### 문제 2 — 개발자

> **빠른 검증이 필요한 개발자가**, PoC나 사이드 프로젝트를 시작하는 상황에서, **초기 보일러플레이트 설정에 반나절 이상을 낭비하는 비효율이 반복된다.**

- 기존 CLI 도구는 설정 자동화 수준이 제한적이다.
- AI 생성 서비스는 기술 스택 자유도가 없거나, 생성 결과물의 품질과 실행 가능성을 보장하지 않는다.

## 3. 목표 상황 (To-Be)

### 비개발자의 경험 변화

- 서비스 아이디어를 자연어로 한 문단 입력한다.
- 화면에서 "요구사항 분석 중 (20%) → 코드 생성 중 (60%) → GitHub 업로드 중 (90%)"이 단계별로 표시되어, 기다리는 동안 무슨 일이 일어나고 있는지 파악할 수 있다.
- 5~10분 후 GitHub clone URL을 받는다. 이 URL을 개발자에게 넘기거나, 클라우드 IDE(Replit, CodeSandbox 등)에 붙여넣어 바로 실행할 수 있다.
- 생성 이력에서 과거 결과물의 URL을 언제든 재확인할 수 있다.

### 개발자의 경험 변화

- 기술 스택(NestJS + React + PostgreSQL 등)을 지정하거나, 기본값으로 바로 생성한다.
- 10분 이내에 ESLint, Prettier, Jest, Docker, GitHub Actions가 모두 설정된 코드베이스를 GitHub 저장소로 받는다.
- `git clone → npm install → npm run dev`를 실행하면 오류 없이 정상 동작한다.
- 바로 핵심 비즈니스 로직 구현에 집중한다.

## 4. 문제 해결 시 기대 효과

### 사용자 관점

| 페르소나 | 기대 효과 |
|---------|----------|
| 비개발자 창업 준비생 | 아이디어에서 데모까지의 시간이 수 주 → 수십 분으로 단축. 개발자 없이도 첫 번째 MVP를 경험 |
| 주니어 개발자 | 보일러플레이트 설정 반나절 → 10분 이내. 포트폴리오 프로젝트 시작 허들 제거 |
| 시니어 개발자/CTO | PoC 검증 사이클 단축. 아이디어가 나온 당일 데모 가능 |

### 비즈니스 관점

- **초기 채택 지표**: 생성 완료 후 clone URL을 실제로 사용한 비율 (clone 클릭률)
- **재방문 지표**: 동일 사용자의 반복 생성 횟수 (서비스가 워크플로에 편입되었는지 여부)
- **바이럴 효과**: 비개발자 사용자가 clone URL을 개발자에게 공유하는 행위 → 개발자 신규 유입 채널
- **가설 검증**: "자연어 입력만으로 실행 가능한 코드베이스 생성" 가설(MVP-scope 가설 1) 검증 시 시장 확장 근거 확보

## 5. 문제의 크기

### 잠재 사용자 규모 추정

> 가정: 아래 수치는 공개 통계 및 시장 보고서에서 발견하지 못해 추론으로 작성했습니다.

**비개발자 세그먼트**

- 한국 기준 스타트업 생태계 종사자(비개발 직군)와 사이드 프로젝트를 준비 중인 비개발자 수는 수십만 명 규모로 추정된다.
- Lovable.dev가 출시 후 단기간에 수십만 사용자를 확보한 것은 이 세그먼트의 실질적인 수요를 방증한다.

**개발자 세그먼트**

- 국내 웹 개발자 수는 30만~50만 명으로 추정된다 (소프트웨어 개발자 전체 약 70만 명 중 웹 개발 비중 기준).
- 이 중 사이드 프로젝트, 포트폴리오, PoC를 주기적으로 만드는 개발자 비율을 20%로 가정하면 6만~10만 명의 잠재 사용자가 존재한다.

**글로벌 관점**

- AI App Builder 시장은 2025년 기준 급성장 중이며, Bolt.new와 Lovable 모두 수백만 월 방문자를 보유한 것으로 알려져 있다.
- mvp-builder의 차별점(GitHub 자동 저장소 생성 + clone 즉시 실행 보장 + 기술 스택 선택)은 기존 서비스의 빈틈을 노리는 포지셔닝이다.

> 가정: MVP 초기 목표 사용자는 국내 개발자 및 기술 친화적인 창업 준비생으로 설정하며, 초도 검증 후 비개발자 세그먼트로 확장한다.
