# 프로젝트 헌법 — AI 기반 자동화 MVP 빌더

> 이 문서는 모든 설계 결정과 개발 활동의 최상위 기준이다.
> 이후 모든 단계의 에이전트와 개발자는 이 원칙을 준수한다.

---

## 프로젝트 비전

**한 줄 비전**: 자연어 요구사항 입력 한 번으로, `docker compose up --build` 한 명령어로 즉시 실행되는 MVP를 사용자 GitHub에 자동 생성한다.

### 핵심 가치

| ID | 키워드 | 설명 |
|----|--------|------|
| V-01 | **소유권** | 생성된 코드는 완전히 사용자의 것이다. 플랫폼 종속성 없이 사용자 GitHub에 직접 저장되며 언제든 독립 개발이 가능하다. |
| V-02 | **실행 가능성** | 단순 코드 스니펫이 아닌, `docker compose up --build` 한 명령으로 즉시 동작하는 완전한 코드베이스를 보장한다. |
| V-03 | **비용 투명성** | 플랫폼 자체 과금 없이 사용자의 Claude API Key를 직접 사용한다. 실제 모델 사용량만 과금된다. |
| V-04 | **기술 선택권** | 기술 스택과 아키텍처를 사용자가 직접 선택하고 확정한다. 플랫폼이 강제하는 기술 제약이 없다. |
| V-05 | **검증 우선** | 코드 생성 전 분석 문서(ERD, API 스펙, 아키텍처)를 먼저 생성하고 사용자 확인을 거친다. |

## 코드 품질 기준

### 언어 및 타입 안전성

- **[결정] 플랫폼 프론트엔드**: Next.js 14+ (App Router), TypeScript strict mode
- **[결정] 플랫폼 백엔드**: NestJS (TypeScript), strict mode 활성화
- **[결정] AI 에이전트 레이어**: Claude Agent SDK (TypeScript) — NestJS 백엔드에 통합
- 모든 `any` 타입 사용 금지. 불가피한 경우 `unknown` + 타입 가드 사용
- 함수 반환 타입은 명시적으로 선언한다

### 코드 스타일

- ESLint + Prettier 기반 자동 포맷팅
- 함수 최대 길이: 50줄 이내 (초과 시 분리)
- 단일 책임 원칙(SRP): 하나의 모듈은 하나의 변경 이유만 가진다

### 복잡도 관리 원칙

- 클래스/모듈 크기: 200줄 이내 권장
- 순환 의존성 금지
- AI 에이전트 파이프라인은 각 단계를 독립 모듈로 분리하여 테스트 가능성을 확보한다

### 에러 처리 원칙

- 외부 서비스 호출(Claude API, GitHub API)은 반드시 try-catch + 구조화된 에러 응답
- 에러는 사용자에게 의미 있는 메시지로 변환하여 반환
- 파이프라인 단계별 실패 지점을 명확히 로깅한다

## 테스트 전략

### 테스트 계층

| 계층 | 범위 | 커버리지 목표 |
|------|------|--------------|
| Unit Test | 순수 함수, 유틸리티, 도메인 로직 | 핵심 비즈니스 로직 80% 이상 |
| Integration Test | **[결정]** API 엔드포인트, 파이프라인 단계 간 연동, DB 연동 | 주요 파이프라인 흐름 100% |

**[결정]** E2E 테스트(브라우저 자동화)는 MVP 범위에서 제외한다. GitHub 연동 등 외부 서비스 검증은 수동으로 대체한다.

### 테스트 필수 대상 (C-TEST-01)

- 요구사항 분석 → 문서 생성 파이프라인 각 단계
- API 엔드포인트 전체 (happy path + error case)
- GitHub 저장소 생성 및 코드 업로드 로직
- Claude Agent SDK 호출 래퍼 (mock 사용)

### 테스트 선택 대상 (C-TEST-02)

- UI 컴포넌트 (핵심 화면 위주)
- 유틸리티 함수

### 모킹 원칙 (C-TEST-03)

- **[결정] TDD 기반 개발 적용**: 테스트 코드를 먼저 작성한 뒤 최소 구현 → 리팩터링
- 외부 API(Claude API, GitHub API)는 mock 사용
- 데이터베이스는 테스트용 PostgreSQL 인스턴스 사용 (mock 금지)

## 보안 원칙

### 인증/인가 (C-SEC-01)

- **[결정]** 사용자 인증: GitHub OAuth 2.0
- 세션 관리: JWT (Access Token + Refresh Token)
- 모든 API 엔드포인트는 인증 필요 (공개 엔드포인트는 명시적으로 표시)

### 민감 데이터 처리 (C-SEC-02)

- **사용자 Claude API Key**: 저장 시 AES-256 암호화 필수. 로그에 절대 노출 금지
- **GitHub Access Token**: 암호화 저장. 만료 시 즉시 무효화
- 개인정보 보존 기간: 회원 탈퇴 후 30일 이내 삭제

### 외부 입력 검증 (C-SEC-03)

- 모든 사용자 입력은 서버 측에서 유효성 검사 (DTO + class-validator)
- 생성된 코드 실행 환경은 사용자 GitHub으로 전달되며, 플랫폼 서버에서 직접 실행하지 않는다

### 비밀 정보 관리 (C-SEC-04)

- 환경 변수로 관리 (`.env` 파일은 Git에 커밋 금지)
- 운영 환경: AWS Secrets Manager 또는 환경 변수 주입

## UX 원칙

### 디자인 철학

- **[결정]** 단계별 진행 UI: 사용자가 파이프라인의 현재 단계를 항상 인지할 수 있도록 설계
- 정보 밀도: 개발자 친화적이되, 비개발자도 핵심 진행 상황을 파악 가능한 수준
- 분석 문서 검토 단계에서 사용자가 수정 피드백을 줄 수 있는 인터랙션 제공

### 접근성 기준

> 가정: WCAG 2.1 Level AA를 목표로 한다. MVP에서는 기본 접근성(시맨틱 HTML, 키보드 탐색)만 보장한다.

### 반응형 전략

- **데스크톱 퍼스트**: 주 사용자가 개발 환경(PC)에서 작업한다고 가정
- 모바일 지원은 MVP 이후 버전에서 고려

### 성능 UX 기준

- AI 코드 생성은 수분 이상 소요될 수 있으므로 **실시간 진행 상황 스트리밍 UI 필수** (SSE 기반)
- API 응답: 일반 CRUD 2초 이내, AI 생성 단계는 스트리밍으로 대체

## 인프라 및 배포 원칙

### 컨테이너화 (C-INFRA-01)

- **[결정]** 플랫폼 자체도 Docker Compose로 로컬 개발 및 배포
- 생성되는 MVP 코드베이스는 반드시 `docker compose up --build` 실행 가능한 형태로 전달

### CI/CD (C-INFRA-02)

> 가정: GitHub Actions 기반 CI/CD. PR 머지 시 자동 테스트 실행, main 브랜치 머지 시 자동 배포.

### 환경 분리 (C-INFRA-03)

| 환경 | 용도 |
|------|------|
| 개발(local) | 개인 개발 환경, Docker Compose |
| 스테이징 | PR 리뷰 및 통합 테스트 |
| 프로덕션 | 실 서비스 운영 |

**[결정]** 클라우드 플랫폼은 AWS를 기준으로 설계한다 (ECS 또는 EC2 기반).

### 모니터링 및 로깅 (C-INFRA-04)

> 가정: MVP 단계에서는 구조화된 로깅(Winston/Pino)과 기본 헬스체크 엔드포인트만 구성한다.
