## 1. MVP 성공 기준

### North Star Metric

> **"월간 생성 완료 건수 (Monthly Completed Generations)"**

이 지표는 mvp-builder의 핵심 가치인 "자연어 요구사항 하나로 즉시 실행 가능한 GitHub 저장소를 만든다"가 실제로 실현된 횟수를 대표한다. 사용자가 서비스에 가입하고, 요구사항을 입력하고, AI 생성 파이프라인을 성공적으로 통과해 clone URL을 받아간 횟수를 카운트한다.

---

### Primary KPI

#### KPI-P-01 생성 성공률 (Generation Success Rate)

| 항목 | 내용 |
|------|------|
| 정의 | 전체 생성 요청 중 `status=completed`로 완료된 비율 |
| 목표 수치 | 90% 이상 |
| 측정 기간 | 출시 후 Month 1 기준 달성, 이후 유지 |
| 현재 Baseline | - |

> 가정: 출시 전 내부 테스트에서 10회 이상 수동 생성 테스트를 수행해 baseline을 측정한다. `MVP-scope.md` 가설 1 검증 기준(90% 이상)과 동일하게 설정했다.

**산출식**: `completed 건수 / (completed + failed + timeout 건수) × 100`

---

#### KPI-P-02 생성 완료 후 clone URL 클릭률 (Clone URL Click-through Rate)

| 항목 | 내용 |
|------|------|
| 정의 | 생성 완료 화면(S-08) 도달 사용자 중 "URL 복사" 또는 "GitHub에서 열기" 버튼을 클릭한 비율 |
| 목표 수치 | 70% 이상 |
| 측정 기간 | Month 1 내 측정, Month 3 기준 목표 달성 |
| 현재 Baseline | - |

> 가정: 경쟁 서비스(Lovable, Bolt.new)의 결과물 사용 전환율 공개 데이터가 없으므로, `problem-statement.md` 비즈니스 관점의 "초기 채택 지표(clone 클릭률)"를 기반으로 70%를 목표로 설정했다. 개발자 페르소나 기준 clone 후 직접 실행하는 행위를 포함한다.

**산출식**: `완료 화면에서 URL 복사 또는 GitHub 버튼 클릭 세션 수 / 완료 화면 도달 세션 수 × 100`

| 항목 | 내용 |
|------|------|

## 2. 지표별 측정 계획

### KPI-P-01 생성 성공률

| 항목 | 내용 |
|------|------|
| 측정 방법 | `generations` 테이블 직접 쿼리. `status` 컬럼 기준 집계. |
| 측정 주기 | 일간 자동 집계 (CloudWatch 대시보드 또는 직접 DB 쿼리) |
| 담당 도구 | 자체 DB 쿼리 (PostgreSQL) / AWS CloudWatch Logs (에러 로그 연계) |

```sql
-- 생성 성공률 쿼리 예시
SELECT
  DATE_TRUNC('day', created_at) AS day,
  COUNT(*) FILTER (WHERE status = 'completed') AS completed,
  COUNT(*) FILTER (WHERE status IN ('failed', 'timeout')) AS failed,
  ROUND(
    COUNT(*) FILTER (WHERE status = 'completed')::numeric /
    NULLIF(COUNT(*) FILTER (WHERE status IN ('completed', 'failed', 'timeout')), 0) * 100,
    2
  ) AS success_rate_pct
FROM generations
WHERE created_at >= NOW() - INTERVAL '30 days'
GROUP BY 1
ORDER BY 1 DESC;
```

### 피벗 조건 및 시나리오

#### 시나리오 A — 생성 성공률 문제 (가설 1 실패)

**조건**: 생성 성공률이 80% 미만으로 지속되고, 실패 원인이 Claude API 품질 또는 생성 결과물 실행 불가 문제인 경우

**대응 방향**:
- 생성 파이프라인 단계별 실패 원인 분석 (어느 단계에서 실패하는지)
- 프롬프트 엔지니어링 개선 (생성 지시 최적화)
- 생성 결과물 검증 단계 추가 (자동 lint 실행, package.json 유효성 검사 등)
- 지원 기술 스택을 검증된 조합으로 제한 (예: NestJS + React만 지원)

---

#### 시나리오 B — 대기 이탈률 과다 (가설 2 실패)

**조건**: 생성 대기 이탈률이 60% 초과이고, 사용자 인터뷰에서 "생성이 너무 오래 걸린다"는 피드백이 다수인 경우

**대응 방향**:
- 생성 소요 시간 단축을 위한 파이프라인 최적화 (병렬 처리 검토)
- 대기 중 "예상 완료 시간" 표시 기능 추가
- 이메일 또는 알림으로 완료 시 통보하는 비동기 방식 도입 (사용자가 대기하지 않아도 됨)
- 간단한 요구사항에 대한 Fast Path 파이프라인 분리

---

#### 시나리오 C — PMF 미달 (월간 활성 생성자 정체)

**조건**: Month 3 기준 월간 활성 생성자가 100명 미만이고, 재방문율이 낮으며, NPS가 10 미만인 경우

**대응 방향**:
- 타겟 세그먼트 재정의: 주니어 개발자보다 시니어/테크 리드 세그먼트 집중 (PoC 요구가 더 명확)
- 가격 정책 도입 검토: 현재 무료 → 크레딧 기반 유료화로 전환해 서비스 가치 재조명
- 공유 기능(URL 공유) 강화로 바이럴 효과 극대화
- 커뮤니티 채널(개발자 슬랙, 오픈채팅방) 직접 참여를 통한 피드백 수집 및 입소문 유도

## 5. 정성적 성공 지표

### 사용자 인터뷰 / 설문 핵심 질문

**개발자 대상 (페르소나 1 — 이태양, 페르소나 2 — 김현우)**
1. "`git clone → npm install → npm run dev` 실행이 오류 없이 성공했나요?" → 가설 1 직접 검증
2. "생성된 코드의 품질(구조, 패턴, 컨벤션)을 1~5점으로 평가한다면?" → 코드 품질 만족도
3. "분석 문서(ERD, API 설계, 아키텍처) 확인이 결과물 신뢰도에 도움이 되었나요?" → 분석 문서 피드백 루프 가치 검증
4. "이 서비스로 아낀 시간이 평균 몇 시간이라고 생각하시나요?" → 가치 제안 수치화
5. "생성된 코드를 그대로 사용했나요, 수정했나요, 버렸나요?" → 결과물 실용성 측정
