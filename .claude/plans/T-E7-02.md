# T-E7-02: PipelineService 상태 머신 및 resume 전략 구현

## Context

현재 `PipelineService.start()`는 BullMQ 잡 등록까지만 구현되어 있고,
`PipelineWorker`의 `handleStart` / `handleFeedback` / `handleConfirm`은 전부 stub.

`confirm` / `feedback` 엔드포인트도 없어서 Phase 2→3 실행 경로가 전혀 없는 상태.

목표: Worker에 Phase 서비스 3개를 연결하고, confirm/feedback HTTP 엔드포인트를 추가해
Phase 1→2→3 파이프라인이 end-to-end로 동작하게 한다.

---

## 구현 계획

### 1단계: ClaudeModule 신규 생성 + PipelineModule import

**`src/claude/claude.module.ts`** 신규:
```
imports:   [S3Module, DockerModule]
providers: [ClaudeAgentService, Phase1Service, Phase2Service, Phase3Service]
exports:   [Phase1Service, Phase2Service, Phase3Service]
```

Phase3Service → DockerSandboxService, S3Service에 의존하므로 S3Module, DockerModule을 ClaudeModule에서 import.
ClaudeAgentService는 Phase 서비스들이 공통으로 쓰는 내부 의존성이므로 export 불필요.

**`src/pipeline/pipeline.module.ts`** 수정:
- `imports`에 `ClaudeModule` 추가 (S3Module, DockerModule은 ClaudeModule이 관리)

> S3Module / DockerModule은 이미 구현됨 (`src/s3/s3.module.ts`, `src/docker/docker.module.ts`)

---

### 2단계: PipelineService에 confirm / feedback 메서드 추가

`pipeline.service.ts`에 AnalysisDocument Repository 주입 추가 후 메서드 2개 구현.

**`confirm(projectId, analysisDocumentId)`**
1. Project 존재 확인 (404)
2. RUNNING 파이프라인 없는지 확인 (409 PIPELINE_ALREADY_RUNNING)
3. `AnalysisDocument.isConfirmed = true` 업데이트
4. `PipelineRun` 신규 생성 (phase=PHASE_2, status=RUNNING)
5. BullMQ `CONFIRM` 잡 등록 — `{ projectId, pipelineRunId }`

**`feedback(projectId, analysisDocumentId, feedbackText)`**
1. Project 존재 확인 (404)
2. RUNNING 파이프라인 없는지 확인 (409)
3. `AnalysisDocument.userFeedback = feedbackText` 업데이트
4. `PipelineRun` 신규 생성 (phase=PHASE_1, status=RUNNING)
5. BullMQ `FEEDBACK` 잡 등록 — `{ projectId, pipelineRunId, feedbackText }`

### 3단계: PipelineController 엔드포인트 추가

`pipeline.controller.ts` 수정:

```
POST /v1/pipeline/:projectId/confirm   → body: { analysisDocumentId }              → 202
POST /v1/pipeline/:projectId/feedback  → body: { analysisDocumentId, feedbackText } → 202
```

inline body 타입으로 처리 (별도 DTO 클래스 불필요).

### 4단계: PipelineWorker 구현

Worker에 추가 주입:
- `Phase1Service`, `Phase2Service`, `Phase3Service`
- `@InjectRepository(PipelineRun)`, `@InjectRepository(Task)`

#### handleStart
```
try {
  await phase1Service.run(projectId)
  update PipelineRun → status=COMPLETED, completedAt=now
} catch(e) {
  update PipelineRun → status=FAILED, errorMessage=e.message
}
```

#### handleFeedback
```
try {
  await phase1Service.run(projectId, job.data.feedbackText)
  update PipelineRun → status=COMPLETED, completedAt=now
} catch(e) {
  update PipelineRun → status=FAILED, errorMessage=e.message
}
```

#### handleConfirm (resume 전략 포함)
```
try {
  // Phase 2: BullMQ 자동 재시도 시 Task 중복 생성 방지 — Tasks 없을 때만 실행
  const count = await taskRepo.count({ where: { pipelineRunId } })
  if (count === 0) {
    update PipelineRun → phase=PHASE_2
    await phase2Service.run(projectId, pipelineRunId)
  }

  // Phase 3: orderIndex 오름차순, status=DONE 태스크 skip, IN_PROGRESS도 재처리
  update PipelineRun → phase=PHASE_3
  const tasks = await taskRepo.find({ where: { pipelineRunId }, order: { orderIndex: 'ASC' } })
  for (const task of tasks) {
    if (task.status === TaskStatus.DONE) continue
    await phase3Service.run(projectId, task.id)
  }

  update PipelineRun → status=COMPLETED, completedAt=now
} catch(e) {
  update PipelineRun → status=FAILED, errorMessage=e.message
}
```

**resume 요약**: DONE 태스크만 skip. PENDING / IN_PROGRESS / FAILED는 모두 재처리.
BullMQ 자동 재시도 시 Phase 2는 Task count로 idempotency 보장.

### 5단계: Unit Test

**`pipeline.service.spec.ts`** 수정 — 케이스 추가:
- `confirm` 정상: isConfirmed 업데이트, PipelineRun PHASE_2 생성, CONFIRM 잡 등록
- `confirm` 404: Project 없을 때 NotFoundException
- `confirm` 409: RUNNING 파이프라인 존재 시 ConflictException
- `feedback` 정상: userFeedback 업데이트, PHASE_1 PipelineRun 생성, FEEDBACK 잡 등록

**`pipeline.worker.spec.ts`** 신규 — 케이스:
- `handleStart` 정상: Phase1Service.run 호출, PipelineRun COMPLETED 갱신
- `handleStart` 실패: Phase1Service.run throw, PipelineRun FAILED 갱신
- `handleFeedback` 정상: Phase1Service.run(projectId, feedbackText) 호출
- `handleConfirm` 정상: Phase2 실행 → Phase3 순서 실행 → PipelineRun COMPLETED
- `handleConfirm` DONE skip: DONE 태스크에 Phase3Service.run 미호출
- `handleConfirm` BullMQ 재시도(count > 0): Phase2Service.run 미호출, Phase3만 실행
- `handleConfirm` Phase3 실패: PipelineRun FAILED 갱신
