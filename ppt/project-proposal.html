사용자 소유권 기반, 실행 가능한 코드베이스 생성 파이프라인

2019920018 김준호

배경과 동기

문제 상황 및 기존 플랫폼의 한계

  • 개인 및 소규모 팀이 아이디어를 검증할 때 발생하는 초기 개발 비용 및 시간 병목 현상
  • 비개발자는 외주 비용 부담, 노코드 툴의 플랫폼 종속성에 직면
  • 개발자는 보일러플레이트 작성 및 테스트, 실행 환경 설정에 반복적으로 시간 소모
  • 기존 AI 빌더들은 토큰/크레딧 과금으로 비용 예측이 어렵고 기술 스택 제약이 존재

프로젝트 목표 및 핵심 동기

  • 자연어 기반 분석 문서를 우선 도출하고, 사용자 GitHub에 직접 실행 가능한 MVP 생성
  • 아이디어 검증까지 걸리는 시간을 단축하여 빠른 시장 테스트 지원
  • 기술 스택과 아키텍처 선택권을 사용자에게 보장
  • Docker 기반 환경 제공으로 로컬 및 클라우드 이식성 확보
  • 플랫폼 내부 종속성을 탈피하고, 사용자의 완전한 코드 소유권 및 후속 개발 보장
  • 플랫폼 과금이 아니라 실제 모델 사용량 기반 구조를 지향해 비용 투명성 확보

프로젝트 핵심 난이도 및 설계 전략

핵심 난이도

  • 단순한 코드 스니펫 생성이 아님
  • 실행 가능하고 후속 개발이 가능한 수준의 MVP를 안정적으로 사용자에게 전달해야 함

고려할 사항 (어려운 이유)

  • 분석 → 피드백 → 생성 → 테스트 → GitHub 업로드로 이어지는 다단계 파이프라인 정합성 확보
  • docker compose up --build 명령어를 통한 즉각적인 실행 가능성 보장
  • 개발자 제어권(아키텍처 자유도)과 자동화 편의성의 균형점 도출
  • GitHub OAuth, 저장소 생성, AI 생성 등 외부 서비스 연동 리스크 제어

설계 전략

  • 모든 기능을 제공하는 범용 App Builder의 전체 구현을 과감히 배제
  • GitHub 직접 전달 + 실행 가능성 보장에만 집중한 MVP 범위로 문제를 제한하여 프로젝트 완성도와 가치를 극대화

경쟁 플랫폼 대비 차별점

비교 항목 기존 플랫폼 (Lovable, Bolt.new 등) MVP-Builder (본 프로젝트)
비용 및 과금 구조 플랫폼 자체 토큰 및 크레딧 기반 과금
(비용 예측이 매우 어려움)		 사용자 API Key 직접 사용
실제 모델 사용량 중심의 투명한 비용 구조
개발 및 실행 환경 브라우저 실시간 실행 등 플랫폼 내부 환경
(선택 가능한 기술 스택 자유도 제한)		 Docker 기반 실행 환경 제공
사용자 주도의 기술 스택 및 아키텍처 선택 가능
결과물 소유권 플랫폼 내부에 머무르거나 연동에 그침
(후속 독립 개발을 위해 별도 정리 필요)		 사용자 GitHub에 Repository 직접 생성
온전한 코드 소유권 보장 및 즉각적인 후속 개발 가능

MVP 생성 파이프라인 구상도

Claude Agent SDK 기반 서버 구성 TDD 기반 개발 방법론 적용
Phase 1. 분석 및 태스크 정의
  • 요구사항 입력: 기술 스택, 아키텍처 등 추가 요건 포함
  • 분석 문서 생성: ERD, API 설계, 아키텍처 초안 제공
  • 피드백 및 확정: 문서 검토 후 승인 또는 수정 요청
  • 태스크 분해: 확정 문서를 기준으로 구현할 기능 단위 정의
Phase 2. TDD 기반 반복 구현
  • 단위 재분해: 성공/실패 케이스, 상태 전이 등 테스트 가능 단위 세분화
  • 테스트 코드 우선 작성: Integration/Unit Test를 먼저 작성하여 기대 동작 명확히 정의
  • 최소 구현 & 리팩터링: 테스트 통과를 위한 최소 코드 작성 후, 구조 정리 및 중복 제거 반복
Phase 3. 통합 검증 및 배포
  • 파이프라인 통합: 분석, 피드백, 생성, 업로드까지 전체 흐름이 끊김 없이 동작하는지 검증
  • 최종 결과물 전달: 사용자 GitHub 저장소에 완성된 코드 업로드
  • 실행 보장: docker compose up --build 기준으로 즉시 실행 가능한 MVP 제공
